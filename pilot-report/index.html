01 / Executive Summary

What changes after the pilot?

Your team stops rebuilding the operating view every morning. The source reports still matter, but the collection, checks, assembly, caveats, and first-pass interpretation run as one workflow.

BeforeReports arrive in inboxes, folders, PMS exports, rate shops, and spreadsheets. A person turns that pile into a morning view.
AfterThe workflow collects the source reports, archives originals, checks completeness, builds the operating brief, and flags what needs attention.
ResultRevenue and operations leaders start from action, not assembly. Missing files and caveats are visible instead of hidden.
02 / The Actual Problem

The report is not the hard part.

Most hotel teams know which numbers belong in the morning view. The fragile part is everything that happens before the output exists.

1
Source reports arrivePMS exports, pickup reports, rate-shop files, forecast sheets, STR or comp-set references, and one-off exceptions.
2
Someone cleans the pileWrong dates, late attachments, duplicate files, renamed columns, and missing reports get handled manually.
3
The view gets assembledNumbers move through spreadsheets, formulas, filters, and formatting before the team can read them.
4
Judgment happens lateThe first half-hour goes to making the report instead of deciding what to do.
5
Trust is implicitIf a value is stale or a file is missing, the final output often looks more certain than it should.
03 / Operating System

The pilot builds the layer underneath the brief.

The deliverable is not just a prettier output. It is a trusted path from source reports to a usable morning operating brief.

IngestionEmail labels, Drive folders, scheduled exports, or approved source locations become the intake queue.
Raw archiveOriginal files are preserved before any transformation, so every number can be traced back.
NormalizationColumns, business dates, property names, and report versions are mapped into a canonical structure.
ChecksMissing inputs, duplicate files, stale values, and date drift are flagged before the team sees the output.
BriefThe final view shows metrics, exceptions, caveats, and first moves in the format the team will actually use.
Run logEvery run records what arrived, what changed, what was missing, and whether the output is complete enough to trust.

04 / Pilot Plan

Four-week pilot. One workflow.

The pilot stays narrow on purpose: one recurring reporting workflow, one owner, defined source reports, defined output, and a handoff the team can keep using.

Week 1

Workflow Map

  • Inventory source reports and expected arrival times.
  • Document owners, manual steps, edge cases, and approval needs.
  • Define the target brief and success criteria.
  • Confirm scope and flat price before build work starts.
Week 2

Build The Source Layer

  • Connect the agreed intake path.
  • Archive raw files before transformation.
  • Map source fields into the canonical report structure.
  • Create missing-file, duplicate, and business-date checks.
Week 3

Ship The Brief

  • Generate the morning operating view.
  • Add caveats, run status, and exception sections.
  • Layer interpretation only after the data is assembled.
  • Run in parallel with the existing manual process.
Week 4

Handoff And Expand

  • Tune the output from daily team feedback.
  • Document rerun, caveat, and support steps.
  • Handoff the operating guide.
  • Identify the next workflow only after the first one holds.
05 / Scope

Exactly what is included.

This is intentionally not a BI rebuild or open-ended strategy engagement. It is a production-minded pilot for one recurring hotel reporting workflow.

Included

  • Workflow review and source report inventory.
  • One defined reporting process across the agreed property set.
  • Automated intake from existing exports, email reports, sheets, or folders.
  • Raw archive, run log, missing-file behavior, and visible caveats.
  • Morning operating brief in the format the team will use.
  • Parallel testing period, handoff notes, and 30-day support window.

Not Included

  • Full enterprise data warehouse rebuild.
  • Unbounded custom software across every department.
  • Autonomous pricing or operational decisions without human approval.
  • Vendor API dependency unless it is already approved and useful.
  • Replacing the team's existing systems on day one.
  • Publishing any client data, names, or proprietary process details.
